# 归档格式规范

服务端启动时扫描两类数据源：视频归档（`archive.db` + 视频文件）和弹幕评论存档
（采集器的输出目录 + 三个任务数据库）。本文描述服务端期望的确切格式。
使用 BiliDanmuComment 采集弹幕评论时其输出天然符合本规范，只有视频归档
需要使用者自行整理。

## 一、视频归档

### 1. archive.db

SQLite 数据库，单表 `videos`：

```sql
CREATE TABLE videos (
    av_number       INTEGER PRIMARY KEY,   -- av 号（纯数字）
    bvid            TEXT    NOT NULL DEFAULT '',  -- 对应 BV 号
    status          TEXT    NOT NULL DEFAULT 'pending',
    title           TEXT    DEFAULT '',
    quality         TEXT    DEFAULT '',
    parts           INTEGER DEFAULT 0,     -- 分P/分集数量，单P 为 1
    file_path       TEXT    DEFAULT '',    -- 见下
    error_msg       TEXT    DEFAULT '',
    confirmed       INTEGER DEFAULT 0,
    manual_override INTEGER DEFAULT 0,
    updated_at      INTEGER DEFAULT 0,
    cover           BLOB    DEFAULT NULL,
    parsed          INTEGER DEFAULT 0
);
```

服务端只读取 `status = 'completed'` 的行，并且只使用
`av_number`、`bvid`、`title`、`parts`、`file_path` 五列。其余列可以留默认值。

### 2. file_path 语义与文件命名

- `parts <= 1`：`file_path` 就是可播放的 MP4 文件完整路径，
  推荐命名 `av{号}.mp4`，例如 `av170001.mp4`
- `parts > 1`：`file_path` 是一个目录，目录内按以下两种规则之一存放：
  - 普通多分P：`av{号}_p{序号}.mp4`，序号从 1 开始，
    例如 `av123_p1.mp4`、`av123_p2.mp4`
  - 番剧（按季一个目录）：`ep{epId:D4}_{标题}.mp4`，epId 为分集的
    episode id 数字（不足四位补零，超过四位原样），
    例如 `ep0267851_第一话.mp4`。服务端从文件名中的 epId 反查分集信息

视频必须是合并完成的单一 MP4 文件（H.264/AAC 或设备可解码的编码），
不支持分离的音视频轨或 DASH 分段。

### 3. 元数据回填

标题、简介、标签、分区、UP 主、发布时间、历史播放与点赞数、封面等元数据
不需要预先准备。服务端在运行时向公开接口按 bvid（番剧按 epId）自动回填，
失败的条目会带原因标记在管理面板中，可手动补录后照常服务。

## 二、弹幕评论存档

### 1. 任务数据库

采集器输出根目录下三个 SQLite 文件（缺失的按不存在处理）：
`collect_history.db`、`collect_current.db`、`collect_comment.db`。
三个库结构相同，单表 `tasks`：

```sql
CREATE TABLE tasks (
    av_number     INTEGER PRIMARY KEY,
    input_token   TEXT DEFAULT '',
    video_type    TEXT DEFAULT '',   -- normal 或 bangumi
    title         TEXT DEFAULT '',
    status        TEXT DEFAULT 'pending',
    danmaku_count INTEGER DEFAULT 0,
    comment_count INTEGER DEFAULT 0,
    output_dir    TEXT DEFAULT '',   -- 该视频存档所在目录的完整路径
    error_msg     TEXT DEFAULT '',
    updated_at    INTEGER DEFAULT 0
);
```

弹幕来源优先使用 `collect_current.db`，同一 av 号只有 history 数据时回退使用。
评论只读 `collect_comment.db`。服务端信任 `output_dir` 列定位存档目录。

### 2. 存档目录内容

每个视频一个目录，内含压缩包（7z 优先，同名 zip 亦可）：

- `danmaku.7z`，内部为 `danmaku/` 目录：
  - 普通视频：`p{分P序号}_cid{cid}.xml`
  - 番剧：`ep{epId:D4}_cid{cid}.xml`
  - XML 为弹幕的原生格式：`<d p="时间,模式,字号,颜色,时间戳,池,用户hash,弹幕id[,权重]">正文</d>`
- `comments.7z`，内部为 `comments/` 目录：
  - 普通视频：`av{号}.json`
  - 番剧：`ep{epId:D4}_aid{aid}.json`

### 3. 评论 JSON 形状

```json
{
  "aid": 170001,
  "total": 12345,
  "replies": [
    {
      "rpid": 0, "oid": 0, "mid": 0, "root": 0, "parent": 0,
      "rcount": 0, "like": 0, "ctime": 0,
      "content": { "message": "评论正文" },
      "member": {
        "mid": 0, "uname": "", "sex": "",
        "level_info": { "current_level": 0 },
        "face": ""
      },
      "reply_control": { "location": "" },
      "sub_replies": [ ]
    }
  ]
}
```

`sub_replies` 为楼中楼数组，元素结构与顶层评论相同，可嵌套。
字段全部为 snake_case。
