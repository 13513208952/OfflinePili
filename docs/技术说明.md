# 技术说明

本文档记录三个组件的技术设计，面向想了解实现或参与修改的读者。
使用方法见仓库根 README，数据格式见 `归档格式规范.md`。

## 总体结构

```
视频文件 + archive.db          （使用者自备，格式见规范）
弹幕评论存档 + collect_*.db    （BiliDanmuComment 产出）
        │  只读
        ▼
服务端 BiliRestart（.NET 9）
  ├─ ASP.NET Core Minimal API：只读资源接口，单端口 5299
  ├─ 回填服务：向公开接口补齐元数据与封面，结果存自己的 catalog.db
  ├─ Avalonia 管理面板：仅本机桌面使用，与 API 同进程
  └─ USB 直连服务：自动为数据线连接的设备布置 adb reverse 隧道
        │  HTTP（局域网 / USB 隧道 / 公网）
        ▼
客户端 OfflinePili-Client（Flutter，基于 PiliPlus）
  ├─ 离线模式数据源分支：视频/弹幕/评论/元数据改从服务端获取
  ├─ 本地互动存储：点赞、投币、收藏、关注、弹幕、评论、观看历史，只写本机
  └─ 本地推荐引擎：在客户端本机计算，不上传任何行为数据
```

## 服务端

- 全部接口为 GET，没有任何写入路由。客户端的互动数据设计上不出设备，
  服务端也不设收集这些数据的通道
- 接口的 JSON 与二进制形状对齐客户端既有解析代码，因此客户端的
  解析与渲染层零改动：
  - `/api/v1/videos`（目录，含标签、分区、UP 主、历史播放与点赞数等特征字段）
  - `/api/v1/videos/{id}`（详情。stat 发全字段，简介同时以 desc 与
    desc_v2 段落两种形式提供，客户端只渲染后者）
  - `/api/v1/videos/{id}/playurl` 与 `/api/v1/stream/{id}/{cid}`
    （直接吐 MP4 字节流，支持 HTTP Range，进度条拖动依赖于此）
  - `/api/v1/danmaku/{cid}`（弹幕按 protobuf 透传，与客户端既有的
    DmSegMobileReply 解码器使用同一份 proto 定义）
  - `/api/v1/replies/{id}`（评论。存档 JSON 与客户端期望的形状不同，
    在服务端重整：楼中楼字段改名、member.mid 转字符串等）
- 回填扫描：进程启动即跑一轮，之后按配置间隔轮询（数据源可能在网络盘上，
  不依赖文件系统事件）。逐条容错，单个视频失败不影响其余；已成功的不重抓。
  失败条目保留原因，可在管理面板重试、手动补录元数据或永久隐藏
- 番剧与普通视频同样支持：分集依赖文件名中的 epId 反查
- 读 7z/zip 存档使用 SharpCompress，纯托管实现，不依赖外部进程
- 管理面板（Avalonia + Material 主题）与 API 同进程，进程内直接调用核心服务，
  不经网络、无认证面。`--headless` 参数跳过界面，用于无桌面环境
- USB 直连：轮询 adb 检测设备并执行 `adb reverse tcp:5299 tcp:5299`，
  重复执行幂等。adb 依 Apache-2.0 随包分发，也可用 PATH 或配置指定的 adb

## 客户端

- 基于 PiliPlus，GPL-3.0。所有对上游文件的修改都包裹在
  `OFFLINE-NOSTALGIA-MODE BEGIN/END` 注释中，新增文件与改动清单见
  客户端仓库 `FORK_DIFF.md`，便于跟进上游时审阅
- 离线模式开启时：数据源入口函数（playurl、弹幕、评论、详情）顶部分支到
  服务端实现；发弹幕、发评论与点赞等互动改写本地 Hive 存储并返回合成的
  成功结果，上层乐观更新逻辑不变；播放进度心跳、相关推荐、字幕、
  在线人数、第三方打点等出站请求全部跳过
- 连接解析：候选地址 = USB（127.0.0.1，走 adb reverse 隧道）、主地址、
  备用地址，按设置的优先档位逐个探测 `/health`，探测结果缓存 60 秒，
  请求失败即失效重探。流与封面地址由服务端按请求 Host 回写，
  自动与当前所用通道一致
- 本地推荐引擎（无网络参与，全部在设备上计算）：
  - 候选硬过滤：拉黑 UP 主（复用上游黑名单机制）与「不感兴趣」视频先行排除，
    再叠加上游既有的规则过滤（时长、播放量、标题屏蔽词）
  - 画像：Rocchio 相关反馈（1971）。特征取标签、分区、UP 主；
    点赞、投币、收藏权重递增，看完为弱信号，负反馈权重高于正反馈基准，
    避免被后续正反馈稀释；按半衰期做时间衰减
  - 质量先验：贝叶斯加权评分（IMDB 公式），以历史点赞率为质量信号、
    播放量为置信度，小样本向全目录均值收缩
  - 多样性：MMR（1998）贪心重排，在相关度与已选结果的差异度之间取舍
  - 关注 UP 主不进入画像，以固定名额穿插其未看完的视频
- 本地存储注意事项：Hive 落盘读回的容器是无类型的，全部读取路径做
  逐层深转换而非 cast，推荐引擎外层再包降级兜底

## 采集器 BiliDanmuComment

- .NET 9 WPF 应用。弹幕支持常规与历史两种模式，评论独立成道，
  多线程逐 av 采集，请求间隔与并发可配
- 需要在设置中提供 Cookie（历史弹幕接口需要登录态），
  并自动维护匿名风控 cookie，避免匿名接口静默截断
- 产物按 `归档格式规范.md` 的目录布局落盘，7z 压缩（内置 7zr，
  不可用时退化为 zip），任务状态记录在三个 SQLite 数据库中

## 构建

- 服务端（在 `server/` 目录下）：`dotnet build BiliRestart.sln`；发布
  `dotnet publish src/BiliRestart.Admin.Host -c Release -r win-x64 --self-contained`。
  Linux 下可构建并以 headless 运行
- 客户端：Flutter（版本以 pubspec.yaml 为准）。Android
  `flutter build apk --release --split-per-abi`；Windows
  `flutter build windows --release`（需要 nuget.exe 在 PATH 中）。
  Windows 下若依赖检出出现缺文件，多为路径长度限制，
  开启 `git config --global core.longpaths true` 后重新获取
- 采集器：`dotnet build`（net9.0-windows，WPF）
